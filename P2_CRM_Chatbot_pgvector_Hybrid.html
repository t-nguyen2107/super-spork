pgvector Hybrid Search

Extends Phase 1 tool calling with semantic vector search — unlocking fuzzy, similarity-based queries that structured tools can't handle. Everything runs inside the same PostgreSQL instance.

pgvector · HNSW nomic-embed-text Ollama · local CPU PostgreSQL 16

Prerequisite — Phase 1 must be complete. P2 builds directly on top of the existing schema, db.py, tools.py, and chatbot.py from P1. No existing code is deleted — only extended. If you haven't done P1 yet, start there first.

Overview

Architecture

Phase 2 adds two things to the P1 architecture: a contact_embeddings table (with HNSW index) alongside the existing schema, and a new semantic_search_contacts tool the LLM can call. The tool router now decides between structured tool calls (P1, real-time) and vector ANN search (P2, sync delay) based on query intent.

Blue paths = P1 structured tool calls (unchanged). Green path = P2 semantic search (new tool added). Dashed line = background embed sync worker writing vectors into contact_embeddings. Both paths converge on the same Postgres instance — no separate vector database.

pgvector schema — what gets added

Key principle: contact_embeddings is a separate table (not a column on contacts). Re-embedding never touches CRM data. The embed_text column stores exactly what was embedded — essential for debugging retrieval misses. Swap models by truncating and re-running the batch.

Step 1 of 6

Enable pgvector in Postgres

pgvector is pre-installed in the official postgres:16-alpine image from Postgres 16 onwards — you just need to enable the extension. No Docker image change required.

bash — enable extension
docker exec -it crm_pg psql -U crm -d crm -c "
  CREATE EXTENSION IF NOT EXISTS vector;
  SELECT extversion FROM pg_extension WHERE extname = 'vector';
"
# Expected: extversion = 0.7.x or higher
migration_p2.sql — run this once
-- ============================================================
-- P2 migration: add pgvector + contact_embeddings table
-- Run: docker exec -i crm_pg psql -U crm -d crm < migration_p2.sql
-- ============================================================

CREATE EXTENSION IF NOT EXISTS vector;

-- Separate table — re-embedding never touches contacts
CREATE TABLE IF NOT EXISTS contact_embeddings (
    contact_id  INT PRIMARY KEY REFERENCES contacts(id) ON DELETE CASCADE,
    embedding   vector(768)     NOT NULL,
    embed_text  TEXT            NOT NULL,  -- what was embedded (for debug)
    model       TEXT            NOT NULL,  -- e.g. nomic-embed-text
    updated_at  TIMESTAMPTZ     NOT NULL DEFAULT now()
);

-- HNSW index: dynamic, no training step, ~30-40 MB at 86k rows
-- m=16: graph connectivity. ef_construction=64: build-time accuracy.
CREATE INDEX IF NOT EXISTS idx_contact_embeddings_hnsw
    ON contact_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Verify
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'contact_embeddings';
bash — apply migration
docker exec -i crm_pg psql -U crm -d crm < migration_p2.sql
# → CREATE EXTENSION, CREATE TABLE, CREATE INDEX

Why HNSW over IVFFlat? At 86k rows, IVFFlat requires a training step and a fixed number of lists chosen at index time. HNSW is dynamic (insert/update without rebuild), faster at this scale, and needs no training. Memory overhead is ~30–40 MB for 768-dim vectors at 86k rows — negligible.

What a good embed_text looks like

The quality of what you embed directly affects retrieval quality. Don't just embed raw column values — build a rich sentence that captures the full context of each contact.

embed_text format (aim for this)
# Good: rich context, readable sentence
"Alice Nguyen works at TechCorp VN in Ho Chi Minh City, Vietnam.
Industry: saas. Status: active. Tags: vip, enterprise.
Notes: Budget approved. Assigned to sales_alice."

# Bad: raw column dump
"Alice|Nguyen|TechCorp VN|saas|VN|active|vip,enterprise|Budget approved"
Step 4 of 6

Embedding Pipeline

Two new files: embedder.py (calls the Ollama embed API) and sync_embeddings.py (batch embeds all contacts, and handles incremental updates).

embedder.py

embedder.py
import os, requests
from dotenv import load_dotenv
load_dotenv()

OLLAMA_URL  = os.getenv("OLLAMA_BASE_URL", "http://localhost:11434/v1")
EMBED_MODEL = os.getenv("EMBED_MODEL", "nomic-embed-text")

# Ollama embed endpoint (not the /v1 path)
EMBED_URL = OLLAMA_URL.replace("/v1", "") + "/api/embeddings"

def embed(text: str) -> list[float]:
    """Embed a single text string. Returns list of floats."""
    resp = requests.post(EMBED_URL, json={
        "model": EMBED_MODEL,
        "prompt": text
    }, timeout=30)
    resp.raise_for_status()
    return resp.json()["embedding"]

def build_embed_text(contact: dict) -> str:
    """Build a rich sentence from a contact record for embedding."""
    parts = []
    if contact.get("name"):
        parts.append(f"{contact['name']} works at {contact.get('company','unknown')}")
    if contact.get("city") or contact.get("country"):
        loc = ", ".join(filter(None, [contact.get("city"), contact.get("country")]))
        parts.append(f"based in {loc}")
    if contact.get("industry"):
        parts.append(f"Industry: {contact['industry']}")
    if contact.get("status"):
        parts.append(f"Status: {contact['status']}")
    if contact.get("tags"):
        tags = contact["tags"] if isinstance(contact["tags"], list) \
               else contact["tags"]
        parts.append(f"Tags: {', '.join(tags)}")
    if contact.get("notes"):
        parts.append(f"Notes: {contact['notes']}")
    if contact.get("assigned_to"):
        parts.append(f"Assigned to {contact['assigned_to']}")
    return ". ".join(parts) + "."

sync_embeddings.py

sync_embeddings.py
"""
Embeds all contacts (or only stale ones) into contact_embeddings.
Usage:
  python sync_embeddings.py          # full batch (first run)
  python sync_embeddings.py --since 1h  # only contacts updated in last 1 hour
"""
import os, sys, time, psycopg2, psycopg2.extras
from dotenv import load_dotenv
from embedder import embed, build_embed_text, EMBED_MODEL
load_dotenv()

def get_conn():
    return psycopg2.connect(
        host=os.getenv("PG_HOST"), port=os.getenv("PG_PORT"),
        dbname=os.getenv("PG_DB"), user=os.getenv("PG_USER"),
        password=os.getenv("PG_PASSWORD"),
        cursor_factory=psycopg2.extras.RealDictCursor
    )

def sync(since_interval: str = None):
    with get_conn() as conn:
        with conn.cursor() as cur:
            # Fetch contacts that need embedding
            if since_interval:
                cur.execute("""
                    SELECT c.* FROM contacts c
                    LEFT JOIN contact_embeddings e ON e.contact_id = c.id
                    WHERE c.updated_at > now() - %s::interval
                       OR e.contact_id IS NULL
                    ORDER BY c.id
                """, (since_interval,))
            else:
                cur.execute("SELECT * FROM contacts ORDER BY id")
            contacts = cur.fetchall()

        print(f"Embedding {len(contacts)} contacts with {EMBED_MODEL}...")
        ok, err = 0, 0

        for i, contact in enumerate(contacts):
            try:
                text   = build_embed_text(dict(contact))
                vector = embed(text)
                with conn.cursor() as cur:
                    cur.execute("""
                        INSERT INTO contact_embeddings
                            (contact_id, embedding, embed_text, model, updated_at)
                        VALUES (%s, %s, %s, %s, now())
                        ON CONFLICT (contact_id) DO UPDATE SET
                            embedding  = EXCLUDED.embedding,
                            embed_text = EXCLUDED.embed_text,
                            model      = EXCLUDED.model,
                            updated_at = now()
                    """, (contact["id"], str(vector), text, EMBED_MODEL))
                conn.commit()
                ok += 1
                if (i+1) % 10 == 0:
                    print(f"  {i+1}/{len(contacts)} done")
            except Exception as e:
                print(f"  ✗ contact {contact['id']}: {e}")
                conn.rollback(); err += 1

        print(f"Done. {ok} embedded, {err} errors.")
        with conn.cursor() as cur:
            cur.execute("SELECT COUNT(*) AS n FROM contact_embeddings")
            print(f"Total in contact_embeddings: {cur.fetchone()['n']}")

if __name__ == "__main__":
    since = None
    if "--since" in sys.argv:
        since = sys.argv[sys.argv.index("--since")+1]
    sync(since)

Production sync strategy: For 78k contacts, the first batch takes ~2–4 hours on CPU (nomic-embed-text does ~10–15 contacts/sec). Subsequent incremental syncs via --since 15m are near-instant. For truly real-time freshness, add a Postgres trigger that inserts changed contact_id into an embed_queue table and have the sync worker drain that queue every minute.

db.py — add semantic_search_contacts

db.py — append to existing file
from embedder import embed  # add to top of db.py

def semantic_search_contacts(
        query: str,
        country: str = None,
        industry: str = None,
        status: str = None,
        min_value: float = None,  # filter by min deal value
        limit: int = 10):
    """
    Hybrid search: cosine similarity on embeddings + optional SQL filters.
    Returns contacts ordered by semantic similarity to query.
    """
    # 1. Embed the query using the same model
    query_vector = embed(query)

    # 2. Build SQL filters on the contacts table
    clauses, params = [], []
    if country:  clauses.append("c.country = %s");   params.append(country.upper())
    if industry: clauses.append("c.industry = %s");  params.append(industry)
    if status:   clauses.append("c.status = %s");    params.append(status)

    # Optional: filter by contacts that have at least one deal above min_value
    if min_value:
        clauses.append("""c.id IN (
            SELECT contact_id FROM deals WHERE value >= %s)""")
        params.append(min_value)

    where = ("AND " + " AND ".join(clauses)) if clauses else ""

    sql = f"""
        SELECT c.id, c.name, c.email, c.company, c.industry,
               c.city, c.country, c.status, c.tags, c.notes,
               c.assigned_to,
               -- cosine distance: 0 = identical, 2 = opposite
               1 - (e.embedding <=> %s::vector) AS similarity_score
        FROM contacts c
        JOIN contact_embeddings e ON e.contact_id = c.id
        WHERE e.embedding IS NOT NULL
        {where}
        ORDER BY e.embedding <=> %s::vector
        LIMIT %s
    """

    # query_vector appears twice: once for score, once for ORDER BY
    final_params = [str(query_vector)] + params + [str(query_vector), limit]

    with get_conn() as conn:
        with conn.cursor() as cur:
            cur.execute(sql, final_params)
            return cur.fetchall()

Hardening Checklist

  • Schedule sync_embeddings.py --since 15m as a cron job or systemd timer — keeps vectors fresh without full re-embedding. Example: */15 * * * * /path/to/.venv/bin/python /path/to/sync_embeddings.py --since 15m
  • Set SET hnsw.ef_search = 100; at query time for higher recall (default 40). Higher value = more accurate but slower. Test at 40, 64, 100 against your data.
  • Add similarity threshold: filter results where similarity_score < 0.6 and tell the LLM "no relevant contacts found" rather than returning low-quality matches.
  • Pin the model column value and add a check — if EMBED_MODEL changes, flag contacts whose stored model doesn't match so they get re-embedded.
  • For 78k production contacts: run the first batch off-hours. At ~10–15 contacts/sec on CPU with nomic-embed-text, 78k contacts takes ~1.5–2 hours. Consider running on a machine with a discrete GPU (llama.cpp supports CUDA/Metal) to speed up to ~200 contacts/sec.
  • Add the embed_queue table pattern for real-time sync: a Postgres trigger inserts changed contact_id values, the sync worker drains the queue on a tight cron loop.
Architecture notes

Key Design Decisions

Separate table

contact_embeddings is separate from contacts. Re-embedding (model swap, schema change) never touches CRM data. Drop and recreate the embeddings table freely.

embed_text column

Stores exactly what was embedded. Essential for debugging retrieval misses — you can inspect what text produced a given vector without reverse-engineering it.

HNSW vs IVFFlat

HNSW wins at this scale. Dynamic (no training step), inserts work without rebuild. At 86k rows, memory overhead is ~30–40 MB. IVFFlat requires upfront training and degrades as data grows.

Single round-trip hybrid

The embedding <=> $1 cosine distance combined with SQL WHERE predicates in one query. No application-side post-filtering. Postgres handles everything.

No separate vector DB

pgvector runs inside the existing Postgres instance. No Pinecone, Weaviate, or Qdrant to operate. One DB, one backup strategy, one connection pool.

Embedding cost

86k records × ~512 tokens ≈ 44M tokens. ~$4–8 one-time with OpenAI ada-002, or free locally with nomic-embed-text. Incremental updates cost near-zero.

Hybrid query pattern — reference SQL

sql — hybrid query anatomy
SELECT
    c.id, c.name, c.company, c.industry, c.status,
    1 - (e.embedding <=> $1::vector) AS similarity_score  -- cosine similarity
FROM contacts c
JOIN contact_embeddings e ON e.contact_id = c.id
WHERE
    c.status = 'active'                          -- SQL filter (exact)
    AND c.industry = 'insurance'                  -- SQL filter (exact)
    AND c.id IN (
        SELECT contact_id FROM deals WHERE value >= 50000)  -- join filter
ORDER BY
    e.embedding <=> $1::vector   -- ANN distance sort (HNSW used here)
LIMIT 10;

-- $1 is the embedded query vector: '[0.023, -0.041, 0.118, ...]'::vector
-- <=> is cosine distance operator (pgvector)
-- 1 - distance = cosine similarity (0..1 scale)
-- HNSW index kicks in on the ORDER BY clause
