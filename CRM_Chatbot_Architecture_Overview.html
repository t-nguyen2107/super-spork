Section 01

The Five Main Approaches

Before choosing an implementation strategy, here is a breakdown of the five main patterns for connecting a chatbot to a CRM database. Each has a different tradeoff between setup complexity, query accuracy, token cost, and security.

Approach 01

Direct SQL — NL2SQL

The LLM generates SQL from the user query, executes it against Postgres, and returns the structured result. Fastest to set up but exposes the schema and carries SQL injection risk.

Cost: Low–Med Tokens/query: ~500–1,500 Local: ✓
Approach 02

Vector RAG (embeddings)

Embed all contacts and deals, store in pgvector. Semantic similarity search retrieves the top-K relevant records per query. Good for fuzzy "find someone like..." queries.

Cost: Medium Tokens/query: ~1,000–3,000 Local: ✓
Approach 03

Pre-computed summaries

Batch jobs periodically summarise and tag records. The LLM queries the index rather than raw rows. Lowest per-query cost but significant batch infrastructure and staleness risk.

Cost: High upfront, low/query Tokens/query: ~300–800 Local: ✓ (costly batches)
★ Recommended
Approach 04

Hybrid — Vector + SQL

Vector similarity for semantic relevance combined with SQL predicates for structured filters (date ranges, deal value, owner). Both run in Postgres via pgvector — no separate infrastructure.

Cost: Medium Tokens/query: ~800–2,000 Local: ✓
Approach 05 · MVP starting point

Tool / Function Calling

The LLM decides which CRM "tool" to call — search_contacts, get_deal, filter_by_stage. Each tool maps to a safe parameterised query. No raw schema exposure, no SQL injection surface, and it handles 80% of chatbot use cases.

Cost: Low–Med Tokens/query: ~600–1,200 Local: ✓ (needs tool-capable model)

Section 03

Recommendation

Hybrid (Approach 4) — with Tool Calling (Approach 5) as the MVP

Phase 1: Build with tool calling first (2–3 weeks). Define 5–8 explicit CRM tools — search_contacts, get_deals, get_contact_history, etc. Each maps to a safe parameterised Postgres query. No vector infrastructure needed yet, zero schema exposure, and this handles 80% of chatbot use cases on day one.

Phase 2: Add pgvector for fuzzy semantic queries — "find contacts similar to our Singapore clients", "anyone in fintech we haven't contacted recently". The pgvector extension runs inside the same Postgres instance, so no new infrastructure is needed. The tool calling layer from Phase 1 is unchanged — Phase 2 just adds a new semantic_search_contacts tool alongside the existing ones.

Key design principle

Never let the LLM write free-form SQL against production. Either parameterised tools or a read-only replica with a restricted role. The tool schema is the security boundary.

Context window budget

78k contacts at even a single sentence each is too large to pass raw. Always retrieve-then-filter — target 10–20 records passed to the LLM per query turn.

Embedding cost

86k records × ~512 tokens ≈ 44M tokens — roughly $4–8 one-time with OpenAI's embedding API, or free if run locally with nomic-embed-text.

Local stack

Ollama + nomic-embed-text + qwen2.5:14b + Postgres with pgvector. Fully air-gapped if needed — important for insurance data sensitivity.

Section 04

Phase Rationale

The two-phase approach delivers value fast while keeping the architecture incrementally extensible. Phase 1 can be demo-ready in days; Phase 2 upgrades the same codebase without rebuilding anything.

Phase 1 · MVP

Tool Calling

Define an explicit API surface of 5–8 CRM tools with typed parameters and enum constraints. The LLM selects and calls tools rather than generating SQL. Every query hits a parameterised function — no schema exposure, no injection surface.

Covers: contact lookup by name/company/country/industry, deal filtering by stage/owner/value/date, pipeline summary, activity history, multi-hop queries (search contact → get their deals).

Outcome: A fully working chatbot that correctly answers ~80% of CRM queries with real-time data freshness.

Phase 2 · Hybrid

pgvector Semantic Search

Add a contact_embeddings table alongside the existing schema. A background sync worker re-embeds changed records. HNSW index on the vector column for sub-100ms ANN search at 86k rows.

Unlocks fuzzy queries impossible with tool calling alone: semantic similarity, implicit clustering ("companies like Acme"), cross-field relevance. The hybrid query combines embedding <=> $1 cosine distance with SQL WHERE filters in a single round-trip.

Outcome: Full semantic + structured query coverage. Phase 1 tools unchanged — Phase 2 adds one new tool.

Section 06

Diagram 2 — pgvector Schema & Embedding Pipeline

The table structure, HNSW index strategy, and how records flow from a CRM change event through text serialisation into a vector upsert. The contact_embeddings table is separate from contacts so re-embedding never touches core CRM data.

Key design decisions: contact_embeddings is a separate table (not a column on contacts) — re-embedding never touches core CRM data and you can swap models by truncating and re-running the batch. The embed_text column stores exactly what was embedded — essential for debugging retrieval misses. HNSW index (m=16, ef_construction=64) is preferred over IVFFlat at this scale — dynamic, no training step, ~30–40 MB memory overhead for 768-dim vectors at 86k rows. The <=> operator is cosine distance in pgvector — combine with SQL predicates for hybrid queries in a single Postgres round-trip.
