Hardening 1 of 5

Error Handling (3 Crash Points)

P1/P2 had zero try/except blocks. Three places could crash the entire program. P2 ENHANCE wraps all of them.

1a. Ollama API call — connection/timeout errors

src/chatbot/cli.py — Ollama error handling
# chat() function — wrap the LLM API call
try:
    response = client.chat.completions.create(
        model=MODEL, messages=history,
        tools=tool_defs.TOOLS, tool_choice="auto"
    )
except (APIConnectionError, APITimeoutError) as e:
    # LM Studio not running or network issue
    console.print(f"[red]Ollama connection error: {e}[/red]")
    return history  # return gracefully, don't crash
except APIError as e:
    # Model not loaded, context overflow, etc.
    console.print(f"[red]Ollama API error: {e}[/red]")
    return history

1b. JSON parse — malformed tool arguments

src/chatbot/cli.py — tool argument parsing
# Local LLMs often return broken JSON: trailing commas, missing braces
for tc in msg.tool_calls:
    try:
        args = json.loads(tc.function.arguments)
    except json.JSONDecodeError as e:
        console.print(f"[yellow]Malformed tool arguments: {e}[/yellow]")
        args = {}  # fallback: call tool with no args (it will use defaults)

1c. Tool execution — database errors

src/chatbot/cli.py — run_tool wrapper
def run_tool(name, args):
    fn = TOOL_MAP.get(name)
    if not fn:
        return {"error": f"Unknown tool: {name}"}
    try:
        result = fn(**args)
        return [dict(r) for r in result]
    except Exception as e:
        # DB down, bad query, type error — return error to LLM
        return {"error": str(e)}

Key insight: When run_tool returns an error dict, the LLM receives it as a tool result. Most LLMs will then explain the error to the user and suggest alternatives — without any special prompt engineering.

Hardening 2 of 5

Chat History Window

Every turn appends messages to the history list. With tool calls, a single turn can add 5-10 messages (user + assistant + tool_call + tool_result × N). After 5-10 turns, the history exceeds most LLMs' context window.

src/chatbot/cli.py — trim_history function
# Maximum messages to keep (excluding system prompt)
MAX_HISTORY = 20

def trim_history(history):
    # Always keep system prompt at index 0
    if len(history) <= MAX_HISTORY + 1:
        return history
    system = history[0]
    rest = history[-MAX_HISTORY:]  # keep most recent messages
    return [system] + rest

# Called after every tool call batch and every assistant response

Why 20? With tool calls, each "turn" uses ~3-5 messages. 20 messages ≈ 4-6 turns, which fits comfortably in most 4k-8k context windows. Adjust MAX_HISTORY based on your model's context size.

Hardening 5 of 5

LM Studio Embedding Compatibility

P2 assumed Ollama's /api/embeddings endpoint. LM Studio uses the OpenAI-compatible /v1/embeddings format instead. The src/embeddings/embedder.py now handles both.

src/embeddings/embedder.py — LM Studio compatible
# Uses OLLAMA_BASE_URL from .env (e.g. http://127.0.0.1:1234/v1)
BASE_URL   = os.getenv("OLLAMA_BASE_URL", "http://localhost:11434/v1")
EMBED_MODEL = os.getenv("EMBED_MODEL") or os.getenv("EMBEDDING_MODEL", "nomic-embed-text")

# Always use /v1/embeddings (OpenAI-compatible format)
EMBED_URL  = BASE_URL.rstrip("/") + "/embeddings"

def embed(text):
    resp = requests.post(EMBED_URL, json={
        "model": EMBED_MODEL,
        "input": text   # OpenAI format uses "input", not "prompt"
    }, timeout=30)
    resp.raise_for_status()
    data = resp.json()
    # Handle both response formats
    if "data" in data:
        return data["data"][0]["embedding"]  # OpenAI/LM Studio format
    return data["embedding"]                   # Ollama format (fallback)

Two env vars for embedding model: EMBED_MODEL is checked first, then EMBEDDING_MODEL as fallback. This lets you use the same .env across different setups without changing variable names.

Step 6

Model Warmup

The first LLM call is always slow — LM Studio needs to load model weights into GPU VRAM, allocate KV-cache, and warm up CUDA kernels. We add a warmup() function that triggers all of this before the user types their first query.

Why warmup matters

Without warmup, the user's first query pays the cold-start penalty (model loading + KV-cache allocation). With warmup, that cost is paid once at startup — the first real query responds at steady-state speed.

src/chatbot/cli.py — warmup function
def warmup():
    """Pre-load model into GPU cache with a tiny request."""
    console.print("[dim]Warming up model...[/dim]", end=" ")
    try:
        start = __import__("time").time()
        client.chat.completions.create(
            model=MODEL,
            messages=[{"role": "user", "content": "Hi"}],
            max_tokens=1,  # minimal — we only care about loading the model
        )
        elapsed = __import__("time").time() - start
        console.print(f"[dim]done ({elapsed:.1f}s)[/dim]")
    except Exception:
        console.print("[dim]skip (model already loaded)[/dim]")

def main():
    console.print("[bold blue]CRM Chatbot[/bold blue]")
    console.print(f"[dim]Model:[/dim] {MODEL}")
    warmup()  # ← called before the chat loop
    console.print("[dim]Type 'exit' to quit[/dim]\n")
    # ... rest of main()

How it works

The warmup sends "Hi" with max_tokens=1. This forces LM Studio to load the full model weights into GPU memory. The cost is ~0.18s one time. Every subsequent call — including the first real user query — responds at warm speed (~0.16s for simple prompts) instead of cold speed (~0.61s).

Why max_tokens=1

We set max_tokens=1 so the LLM generates only 1 token. We don't care about the response — we only want the side effect of loading the model. This minimizes GPU compute waste.

Connection Pooling

Currently, every query opens a new TCP connection. With semantic search (which embeds the query + runs SQL), that's 2 connections per search. At scale, add connection pooling:

src/db/connection.py — optional: connection pool
from psycopg2.pool import SimpleConnectionPool

# Create a pool with min=1, max=5 connections
_pool = SimpleConnectionPool(
    1, 5,
    host=os.getenv("PG_HOST"), port=os.getenv("PG_PORT"),
    dbname=os.getenv("PG_DB"), user=os.getenv("PG_USER"),
    password=os.getenv("PG_PASSWORD"),
    cursor_factory=psycopg2.extras.RealDictCursor
)

def get_conn():
    return _pool.getconn()

# Remember to call _pool.putconn(conn) after use

Not included in P2 ENHANCE. Connection pooling adds complexity. For a learning project with single-user CLI usage, the current per-query connection is fine. Add pooling when you move to a web API or multi-user scenario.

def validate_args(name, args):
    # Ensure contact_id is a positive integer
    if "contact_id" in args:
        try:
            args["contact_id"] = int(args["contact_id"])
            assert args["contact_id"] > 0
        except (ValueError, AssertionError):

Limit Cap

The limit parameter has no upper bound. If the LLM passes limit: 999999, it could fetch the entire table. Add a hard cap in src/db/connection.py:

src/db/connection.py — optional: cap limit
def search_contacts(name=None, ..., limit=10):
    limit = min(limit, 100)  # hard cap: never return more than 100 rows

Also update src/chatbot/tools.py to add "maximum": 100 to the limit parameter schema — this tells the LLM not to request more than 100.

Advanced

Smart Embedding Sync

When deals or activities change, the contact's embedding becomes stale — it was built from old data. The --smart flag detects contacts affected by related data changes and re-embeds them with enriched text that includes deal summaries and recent activities.

Why it matters: A contact's semantic search result should reflect their current deal pipeline and recent interactions, not just their static profile. Without smart sync, a contact with a new $500k negotiation deal would still be found by their old embedding text.

Usage

terminal — smart sync commands
# Smart sync: detect contacts affected by deal/activity changes (lookback=2h)
python -m src.embeddings.sync --smart

# Smart sync with custom lookback window
python -m src.embeddings.sync --smart --since 24h

# Other modes still work:
python -m src.embeddings.sync              # only unembedded contacts
python -m src.embeddings.sync --since 1h   # contacts updated recently
python -m src.embeddings.sync --force      # re-embed everything

How it works internally

src/embeddings/sync.py — fetch_smart() query
-- 5-way UNION: find all contacts affected by any data change
SELECT c.id FROM contacts c
LEFT JOIN contact_embeddings e ON e.contact_id = c.id
WHERE e.contact_id IS NULL          -- 1. no embedding yet

UNION
SELECT c.id FROM contacts c
JOIN contact_embeddings e ON e.contact_id = c.id
WHERE c.updated_at > e.updated_at  -- 2. contact updated after embed

UNION
SELECT id FROM contacts
WHERE updated_at > now() - %s     -- 3. recently updated contacts

UNION
SELECT DISTINCT contact_id FROM deals
WHERE updated_at > now() - %s     -- 4. deals changed

UNION
SELECT DISTINCT contact_id FROM activities
WHERE created_at > now() - %s     -- 5. new activities

When to use each mode: Use --smart for regular maintenance (e.g. cron job every 2h). Use --since 1h for quick touch-ups. Use --force only when changing the embedding model. Use default (no flags) for initial bulk embedding.

Architecture notes

Design Decisions

Error → LLM → User

Tool errors are returned as dicts to the LLM, not thrown. The LLM reads the error message and explains it naturally to the user. No special error UI needed.

Trim, don't summarize

We trim old messages instead of summarizing them. Summarization adds complexity (extra LLM call, latency, potential hallucination). For a CLI chatbot, losing old context is acceptable.

One URL for chat + embed

Both chat and embedding use OLLAMA_BASE_URL. No separate embed endpoint config. LM Studio serves both on the same port — simplifies setup.

OpenAI format everywhere

src/embeddings/embedder.py uses "input" (OpenAI) not "prompt" (Ollama). LM Studio requires OpenAI format. The fallback parser handles both response shapes.

Warmup, not lazy-load

We warm up eagerly at startup with max_tokens=1 instead of letting the first real query pay the cold-start penalty. The 0.18s overhead is invisible to the user (they see "Warming up model..." in the startup banner). Without warmup, the first query with tools takes ~1.5s; with warmup, it takes ~0.3s.

Optional hardening

Connection pooling, input validation, LIKE escaping, and limit caps are documented but not implemented. They're noted as "Advanced" so learners can add them incrementally.

Enriched vs base embeddings

Smart sync (--smart) enriches embedding text with deal summaries and recent activities. Base mode uses contact fields only. This is a tradeoff: enriched embeddings are more useful for semantic search but require extra DB queries per contact. Use enriched for maintenance runs, base for bulk initial embedding.

No new tools

P2 ENHANCE doesn't add new LLM tools. It only hardens existing code. New features (web UI, streaming, auth) belong in P3.

Future

What's Next — Phase 3: AI Deal Intelligence Agent

P2 ENHANCE makes the chatbot robust with exact queries + semantic search. Phase 3 takes the next leap: turning the CRM from a "data container" into a "proactive advisor" using Multi-Agent RAG.

Why P3 matters

CRM data lives in two forms: structured (deal stage, contract value, contact status) and unstructured (call logs, negotiation emails, meeting notes). P2 ENHANCE handles both separately. P3 combines them into a single Hybrid Search + Agentic RAG pipeline that can reason about deal health, detect risks early, and recommend next actions.

P3 Architecture: Agentic RAG Flow

Instead of a single LLM call → tool → response, P3 chains multiple specialized agents in a stateful workflow.

Tech stack additions for P3

LangGraph

Stateful agent orchestration with built-in checkpointing. Each agent is a node in a graph. State (query, filters, retrieved deals, analysis) flows between nodes automatically.

Deal embeddings

New deal_embeddings table stores vector representations of interaction notes. Enables semantic risk detection — "find deals where client sounds hesitant about pricing".

Replace CLI with Streamlit or FastAPI + React. Show each agent's progress in real-time: "Analyzing intent... Retrieving deals... Diagnosing risks... Building strategy."
